Day 03 —
SQLAlchemy ORM >> A Deeper Dive



By : admin 24-07-2022

In order to efficiently manage a database, or write an app that'd be constantly interacting with a database, it is important to solidify the concepts that revolves around understanding the state of your database, because thinking in terms of an object's state will continually be valuable in the development of a dynamic web application.

Today, being the 3rd day of my journey to becoming a full stack developer, I dived deeper into learning about:
— The SQLAlchemy Object Life:
- The various stages of transactions¹ within a session:²
  • Transient: This is the the first stage of transaction. It defines the existence of an object, but not (yet) attached to a session or database.
  • Pending: This defines the attachment of an object to a session, but has not been permanently committed to a database. At this stage, we can still undo/rollback any work that has been done back to the 'transient' stage as long as a 'flush' hasn't occurred.
  • Flushed: At this stage, all actions/works(pending changes) are then translated into SQL commands that are ready to be manually committed to the database.
  • Committed: This is when you finally and manually commit your works(pending/flushed) to the database for a permanent persistence.
Querying Objects:
— Model.query db.Model.query: offers us the ability to query objects by generating SELECT statements that queries and returns slices of data from our database.
— Query Method chaining...
— Ways of accessing query objects on a Model:
  • Directly - MyModel.query
  • Indirectly -db.session.query(MyModel)
— Some useful query methods:
  • .query.all()
  • .query.first()
  • .query.order_by()
  • .query.limit()
  • .query.get()
  • .query.delete()
  • .query.join()
  • .query.count()
  • .query.filter_by()
  • .query.filter() — similar to .query.filter_by(), but you specify attributes on a given model instead. It is more flexible and especially useful when querying from a joined table where you want to filter by attributes that span across multiple models.
¹A transaction can be referred to as a unit of work created within a session

²A session is the interval of action between an opened and closed connection.
